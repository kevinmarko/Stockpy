# Architecture: Signal & Calculation Engines

> Part of the split-out `CLAUDE.md` System Architecture reference. See [../../CLAUDE.md](../../CLAUDE.md) for the index.

- **processing_engine.py** — vectorized fundamental valuations (Graham Number, Gordon Fair Value) and technical indicators (RSI, MACD, Aroon, ATR, SMA). Iterative loops/`.iterrows()` over price series are disallowed; everything must be expressed as pandas/numpy vector operations. **Tier 2.3 Phase 3 wiring**: `calculate_fundamental_metrics()` initialises a `HistoricalStore` instance when `settings.HISTORICAL_STORE_ENABLED=True` and calls `get_fundamentals(ticker, max_age_days=settings.FUNDAMENTALS_REFRESH_DAYS)` to serve cached fundamentals or write fresh ones; the typed return dict is overlaid onto the raw `info` dict before existing downstream calculations (no key-mapping changes in downstream code). `HistoricalStore` is imported lazily (inside the method body) to avoid circular imports. **Rolling beta (2026-07):** module-level `calculate_rolling_beta(price_df, spy_df, window=60) -> pd.Series` computes a TIME-VARYING beta (`.rolling(window).cov(spy_returns) / .rolling(window).var()`, inner-joined by date) — distinct from the existing STATIC single-window `Beta` column (`data/yahoo_fundamentals.py`, surfaced via `config.COLUMN_SCHEMA`), which stays unchanged. Naturally lookahead-free by construction (verified by two perturbation tests in `tests/test_indicators_lookahead.py` — perturbing either the ticker or the SPY series after a cutoff leaves the beta value at that cutoff unchanged); the first `window` rows are `NaN`, never fabricated. No new `COLUMN_SCHEMA` entry (a rolling series doesn't fit one dashboard cell) — surfaced instead as an on-demand chart in the GUI Analytics tab (`gui/panels/analytics.py::_render_rolling_beta_chart`, computed from `HistoricalStore`-cached bars for the selected symbol and SPY).
- **macro_engine.py** — macroeconomic regime detection/gating. `MacroEngine.__init__` now constructs a persistent `regime.hmm_regime.HMMRegimeDetector(n_states=3, retrain_freq_days=7)` instance (`self._hmm_detector`). `compute_hmm_risk_on_probability(spy_price_df)` builds the 4-feature matrix via `regime.hmm_regime.build_feature_matrix()` and returns the HMM's `risk_on_probability` second opinion — or `None` (never fabricated) if SPY history, macro history, or the aligned feature matrix (`< HMM_MIN_FIT_ROWS=100` rows) is insufficient, or if fit/predict raises (logged, never propagated — a statistical second opinion failing must never crash the primary rules-based pipeline). **Tier 2.3 Phase 3 wiring**: when `settings.HISTORICAL_STORE_ENABLED=True`, `compute_hmm_risk_on_probability` routes the VIXCLS and T10Y2Y series through `HistoricalStore.get_macro('VIXCLS')` / `get_macro('T10Y2Y')` (with `data_engine=self.data_engine` as the top-up source) instead of calling `self.data_engine.fetch_macro_history()` directly. On any HistoricalStore failure the code falls back to the direct `DataEngine.fetch_macro_history()` call (pre-Phase-3 behavior). The single-snapshot `_build_macro_dto` path (current-state reads for the kill-switch) is NOT routed through HistoricalStore — only the HMM's historical series benefit from DB caching.
- **regime/hmm_regime.py** — Gaussian HMM regime detector (Hamilton 1989), a statistical second opinion to the rules-based regime in `macro_engine.py`. `build_feature_matrix(spy_price_df, vix_series, yield_curve_series)` builds the 4-feature matrix (`spy_return`, `realized_vol_20d`, `vix_level`, `yield_curve_spread`), normalizing all three inputs' indices to timezone-naive, time-stripped dates before aligning (yfinance is often tz-aware/intraday-stamped; FRED is naive/midnight — without this, misaligned sources silently produce an all-NaN join). `HMMRegimeDetector(n_states=3, retrain_freq_days=7)`: `fit(features_df)` refits only if `retrain_freq_days` have elapsed since the last real fit (expanding window is the caller's responsibility — pass progressively more history each call); within that window, repeated `fit()` calls are no-ops, which is exactly what `tests/test_hmm_no_lookahead.py` exercises. `identify_states_by_vol()` sorts states by fitted diagonal-covariance variance ascending and labels `["bull", "sideways", "bear"]`. `predict_proba(features_df)` returns FORWARD (filtered) probabilities at the LAST ROW only of whatever's passed in — never Viterbi/smoothing — via the mathematical identity that a smoothed posterior's last row in any sequence equals pure forward filtering (the backward pass is seeded with `beta_T=1` at the sequence's own terminal step, so there is no "future" to smooth with). Callers MUST slice their feature frame to end exactly at the date they want a probability for; the API structurally cannot return an interior date's probability.
- **technical_options_engine.py** — options-specific technical metrics (IV rank, GARCH vol, options IV edge). **Premium directive helper**: top-level `build_premium_directive(symbol, bars, *, spot_price, is_stale=False, target_dte=30, macro_dto=None, vrp=None, risk_free_rate=settings.RISK_FREE_RATE) -> dict` fuses GJR-GARCH σ, realized-vol IVR proxy, Aroon+Coppock trend bias (via the deterministic `_determine_trend_bias` rule), full ATM Black-Scholes Greeks, and the deterministic strategy directive from `OptionsPricingRecommender.generate_strategy_pricing_matrix` into a single hydrated row (Symbol, Price, Stale, Sigma_GARCH, IVR_Proxy, Aroon_Oscillator, Coppock_Curve, Trend_Bias, Strategy, Action, Net_Premium, Realizable_Daily_Theta, ATM_{Delta,Gamma,Vega,Theta_Daily}, Short_/Long_Strike, Short_/Long_Delta, Legs, Integrity_OK, Integrity_Issues). NaN — never fabricated zeros — when a primitive can't be computed (CONSTRAINT #4). Centralised here (not duplicated in `gui/panels.py`) so the Gravity AI Review Suite exercises the same code path the GUI does. **Integrity validator**: `validate_directive_integrity(directive, *, delta_tolerance=0.05, strike_grid=STRIKE_GRID_USD=0.50) -> {"ok": bool, "issues": [str], "checks": [dict]}` asserts every leg strike lies on the `$0.50` exchange grid and (when a `Delta` is present on the leg dict) the resolved BS delta is within `delta_tolerance` of the conventional target in `EXPECTED_DELTA_TARGETS` (Put Credit Spread ±0.30/±0.15, Iron Condor ±0.16/±0.05, etc.). Cash/Wait directives pass trivially; Iron Condor legs omit `Delta` by engine convention and the validator skips (never fabricates) that check.
- **forecasting_engine.py** — ARIMA, Monte Carlo, Holt-Winters, Prophet, and CNN-LSTM forecasts. Deterministic trend params are enforced (`trend='t'` for ARIMA, `trend='add'` for Holt-Winters) and Monte Carlo uses structural drift (μ − 0.5σ²) to prevent naive flatline projections. The CNN-LSTM model implements direct multi-horizon forecasting, trains once per ticker, and uses strict train-only scaler fitting and sequence creation to eliminate lookahead bias. **GARCH → Monte Carlo sigma (2026-07):** the Monte Carlo σ is now sourced from the GJR-GARCH(1,1) estimator (`technical_options_engine.estimate_gjr_garch_volatility`, forward-looking + fat-tailed) via the new `ForecastingEngine._estimate_daily_sigma(history_df, fallback_daily_sigma)` helper, instead of a naive historical log-return stdev — making the Monte Carlo bands regime-responsive (they widen in turbulent vol regimes and tighten in calm ones). **SCALE RULE (do NOT "fix"): GARCH returns ANNUALIZED vol; Monte Carlo needs DAILY, so `_estimate_daily_sigma` divides by `sqrt(252)` — mandatory, because `run_monte_carlo`'s auto-normalize guard only inspects `mu`, so it cannot detect/correct an annualized σ.** Degrades to the historical daily std (never raises) when GARCH is unavailable / `history_df is None` / `< 22` rows / the flag is off. Behind `settings.FORECAST_USE_GARCH_SIGMA` (default `True`). **Prophet → ensemble (2026-07):** `run_prophet_forecast`'s h=30 output (previously computed but discarded from the blend) now enters `model_forecasts["prophet"]` at the h=30 horizon ONLY and is folded into `_blend_with_skill`'s static branch as a backward-compatible overlay `final = base*(1-w) + prophet*w`, `w = settings.FORECAST_PROPHET_WEIGHT` (default 0.25, clamped to [0,1]). Prophet still runs at most once per call; Prophet-absent behavior is byte-identical to the pre-overlay blend. **Fit-once hot-loop refactor (2026-07):** `run_arima`/`run_holt_winters_grid_search` are split into fit-once + forecast helpers (`run_arima_fit`/`forecast_from_arima_fit`, `run_holt_winters_fit`/`forecast_from_hw_fit`); `generate_forecast` now fits ARIMA and Holt-Winters exactly ONCE per ticker (both fits are horizon-independent) and reuses each fitted model across the target-days forecast AND every horizon in `[10,30,60,90]`, collapsing ~5 ARIMA + ~12-15 HW statsmodels fits/ticker down to 1 + 3. Output is byte-identical to the pre-refactor per-horizon path (verified by shim-equality + fit-count tests in `tests/test_forecasting_engine.py`); the old `run_arima`/`run_holt_winters_grid_search` names are retained as back-compat shims (fit-once + forecast-once internally). CNN-LSTM (direct multi-step, trained once) and Prophet (h=30, run once) were already single-fit and are unchanged. **GARCH double-fit elimination (2026-07):** `generate_forecast` gained `precomputed_garch_annual_vol: Optional[float] = None`; when supplied, `_estimate_daily_sigma` uses `precomputed_garch_annual_vol / sqrt(252)` (the same SCALE RULE) instead of running a SECOND internal GJR-GARCH fit. `main_orchestrator.py` (~line 410) passes `dashboard_df['GARCH_Vol']` — the per-ticker GJR-GARCH vol it already fit once earlier in the cycle (~line 300) — so no ticker is GARCH-fit twice per cycle; `main.py`/`engine.advisory` callers pass nothing (default `None` → unchanged internal fit). **Skill-weighted blend now operator-activatable (2026-07-09):** the inverse-RMSE `forecasting.forecast_tracker.ForecastTracker` (which weights ARIMA/Monte Carlo/Holt-Winters/CNN-LSTM by recent realized accuracy instead of fixed fractions, in `_blend_with_skill`'s skill branch) is wired behind `settings.FORECAST_SKILL_WEIGHTING_ENABLED` (default `False`, mirroring the `FORECAST_USE_GARCH_SIGMA` opt-in convention). When `True`, a persistent `ForecastTracker` (self-provisioning its `forecast_errors` table in `quant_platform.db`, no migration) is injected into every `ForecastingEngine` construction — `main_orchestrator.py` (both the `EngineContext.build` path and the `run_pipeline` fallback) and `engine/advisory.py`; when `False` (the default) `tracker=None` → byte-identical static blend. `FORECAST_SKILL_WINDOW_DAYS` default was raised `60 → 180` (it MUST exceed the 90-day max horizon: a 'completed' h=90 row needs `forecast_ts ≤ now-85d` while the window only counts `forecast_ts ≥ now-WINDOW`, so 60 left h=60/h=90 with mutually-exclusive eligibility bands and they could never warm up). Both are in `gui/env_io.py` `ALLOWED_KEYS`; new uplift backtest `tests/test_forecast_skill_uplift.py` (slow-marked, opt-in). **New env vars:** `FORECAST_USE_GARCH_SIGMA` (bool, default `True`), `FORECAST_PROPHET_WEIGHT` (float, default `0.25`), `FORECAST_SKILL_WEIGHTING_ENABLED` (bool, default `False`), `FORECAST_SKILL_WINDOW_DAYS` (int, default `180`). **Connection reuse (2026-07, PR A hot-path):** `ForecastTracker` previously opened a fresh `sqlite3.connect(...)` (+ WAL PRAGMA) on every call (~12/ticker/cycle); it now holds ONE cached connection (`self._conn` via `_get_conn()`, `check_same_thread=False`, WAL + `busy_timeout=5000`) guarded by `self._lock` (`threading.Lock`) so every method reuses the handle and it is safe under the orchestrator's parallelized advisory loop; `_ensure_table()` still uses a short-lived connection so the db_path-swap error tests keep passing. **CNN-LSTM + Prophet model persistence (2026-07, PR C, opt-in via `settings.FORECAST_MODEL_PERSISTENCE_ENABLED`, default `False`):** both models were previously retrained from scratch on EVERY `generate_forecast()` call per ticker per cycle — the CNN-LSTM's own docstring already flagged this as unable to scale, and it is by far the dominant per-cycle CPU cost. `forecasting/model_persistence.py` (new module) provides dependency-light `artifact_path(prefix, ticker, suffix)` / `is_fresh(path, retrain_days)` / `touch(path)` helpers — ONE fixed (non-dated) artifact per ticker under `ml/models/forecast_cache/` (mtime IS the staleness clock; simpler than the glob-latest dated-history convention `ml/lgbm_ranker.py`/`ml/meta_labeling.py` use, since only the most recent fit ever matters for a live forecaster). `run_prophet_forecast(..., ticker=None)`: when a fresh (`< FORECAST_MODEL_RETRAIN_DAYS`, default 7) `prophet_{TICKER}.pkl` exists, `model.fit()` is skipped entirely — only `.predict()` runs on the pickle-loaded model; otherwise fits fresh then pickles the fitted model. `run_cnn_lstm_forecast(..., ticker=None)`: same pattern but split across three artifacts — `cnn_lstm_{TICKER}.keras` (native Keras `model.save()`/`tf.keras.models.load_model()`, not pickled) plus `cnn_lstm_{TICKER}_scaler_x.pkl` / `_scaler_y.pkl` (the train-only-fit `MinMaxScaler`s, pickled) — a cache hit skips the 50-epoch `Sequential(...).fit()` entirely and just transforms the latest lookback window through the loaded scaler + model. **Scoped to the direct multi-step path only** (`days_forward is None`, the path `generate_forecast` actually uses) — the single-horizon `days_forward`-set path is intentionally left untouched (out of scope) rather than handling a horizon-count mismatch between a cached multi-step model and a single requested horizon; a defensive `cached_model.output_shape[-1] != len(horizons)` check exists anyway and falls through to a fresh fit on mismatch. `generate_forecast` threads `ticker=persist_ticker` (the already-existing `symbol` extraction, `None` when the row carries no real `Symbol`/`Ticker` so ad-hoc/test calls never touch the on-disk cache) into both calls. Behavior-preserving BETWEEN retrains (same fitted weights → same forecast for repeated calls within the retrain window) — only WHEN a fit happens changes; any load/inference failure (missing library, corrupt/unreadable artifact, insufficient rows, horizon mismatch) degrades to a fresh fit, never raises (CONSTRAINT #6). When the flag is `False` (the default) every call retrains from scratch exactly as before persistence existed — matches the `FORECAST_USE_GARCH_SIGMA`/`FORECAST_SKILL_WEIGHTING_ENABLED` opt-in convention. **New env vars:** `FORECAST_MODEL_PERSISTENCE_ENABLED` (bool, default `False`), `FORECAST_MODEL_RETRAIN_DAYS` (int, default `7`, mirrors `regime/hmm_regime.py`'s `HMMRegimeDetector(retrain_freq_days=7)` convention). Both are in `gui/env_io.py` `ALLOWED_KEYS` and the Settings Manager layout. **Forecast reliability curve (2026-07):** `ForecastTracker.get_forecast_reliability_curve(symbol=None, horizon_days=None, n_bins=10, min_per_bin=3) -> pd.DataFrame` bins the realized percent error `(actual_price - forecast_price) / actual_price` over completed `forecast_errors` rows (`actual_price IS NOT NULL`) into fixed `[-0.5, 0.5]` buckets per `(model_name, horizon_days)` group — a genuinely distinct concept from `evaluation_engine.py`'s `calibration_curve()` (conviction-vs-win-rate from closed trades, not forecast accuracy). Bins with fewer than `min_per_bin` rows get `mean_pct_error=NaN` (never fabricated — CONSTRAINT #4); empty/no-data returns the correct empty schema, never raises (CONSTRAINT #6). Surfaced as a new sub-section in the GUI Observability tab's Forecast Skill panel (`gui/panels/observability.py::_render_observability_forecast_skill`).
- **strategy_engine.py** — signal generation with calibrated momentum/trend weights and ATR-based volatility corridors (buy zone / hold range / trim-stop). Uses an Aroon Oscillator chop-filter to suppress false-positive MACD signals. Position sizing (`_calculate_kelly_sizing`) is volatility-targeting + estimated-p fractional Kelly (see `sizing/`) — no longer a score-derived formula. **Dedicated sell-side range (`apply_sell_side_range`):** alongside the legacy single-corridor `apply_tactical_ranges` (`buyRange`), `evaluate_security()` now ALWAYS emits a first-class `sellRange` string for every Action Signal. Active long signals (STRONG BUY/BUY/HOLD) yield `"Sell Zone: $LO - $HI | Stop @ $STOP"` where LO=`price + 1.5*ATR`, HI=`max(price + 3*ATR, forecast_price)` (forecast wins only when >ATR ceiling — never fabricated when forecast_price=0), STOP=`chandelier_long` (fallback `price - 2.5*ATR`, clamped ≥ $0.01). RISK REDUCE / unknown signals fail closed to `"Sell Now @ market | Stop @ $STOP"` (chandelier_long, fallback `price - 1.0*ATR`). Lookahead-free: consumes only the already-causal ATR, Chandelier Exit, and `Forecast_30` already flowing into `evaluate_security`.
- **sizing/** — Position sizing package: the single source of truth for "Kelly Target" sizing, replacing two divergent arbitrary score-derived win-probability formulas that previously lived in `strategy_engine._calculate_kelly_sizing` and a `main_orchestrator.py` call site into `evaluation_engine.calculate_kelly_target`.
  - **POSITION SIZING SINGLE SOURCE OF TRUTH (sizing/, strategy_engine.py):**
"Kelly Target" MUST be computed via `StrategyEngine._calculate_kelly_sizing(realized_vol, strategy_id=None)`, which returns a `(weight: float, path_tag: str)` tuple.

**Stage 1.7 — Per-Strategy Bootstrap-Conservative Kelly (current):**
- When `strategy_id` is provided, delegates to `sizing.kelly.kelly_sizing_for_strategy(transactions_store, strategy_id, realized_vol)` which:
  1. Calls `estimate_win_rate_and_payoff_per_strategy()` to filter closed trades to `strategy_id`.
  2. Cold start (< 30 per-strategy trades): falls back to `volatility_target_weight(realized_vol)`, tagged `"vol_target_fallback"`. ALWAYS explicitly logged.
  3. Warm path: runs `bootstrap_kelly_confidence(returns, n_bootstraps=1_000)` and returns the **5th-percentile** Kelly fraction — the "epistemic humility" sizing (conservative until the edge estimate is stable at > ~200 trades). Tagged `"bootstrap_kelly_5th_pct(n=...,k5=...,k50=...,k95=...)"`.
- When `strategy_id` is None (backward-compatible): uses the global-pool aggregate `estimate_win_rate_and_payoff()` point estimate. Tagged `"aggregate_kelly"` or `"vol_target_fallback"`.
- Either path is clamped to `settings.MAX_POSITION_WEIGHT=1.0` in `StrategyEngine._calculate_kelly_sizing()` itself; `evaluate_security()` then composes the regime and meta-label multipliers and re-clamps via `sizing.position_sizer.size_position()` — see the dedicated `sizing/position_sizer.py` bullet below for the full ordered pipeline and new guardrail telemetry.
- `evaluate_security()` accepts `strategy_id: Optional[str] = None` and threads it down to `_calculate_kelly_sizing()`.
- Do NOT reintroduce a score/sortino/edge_ratio-derived win-probability formula at any call site.

**Meta-Label Probability — Stage 4 Scaffold:**
- `SignalOutput.meta_label_proba: float = 1.0` — Lopez de Prado meta-label probability, always 1.0 until Stage 4.
- `SignalAggregator.aggregate()` returns a **6-tuple**: `(final_score, score_log, warnings, details, outputs, meta_label_composite)`. The 6th element is the geometric mean of active modules' `meta_label_proba` values. Always 1.0 currently (no-op).
- `StrategyEngine.evaluate_security()` multiplies the Kelly Target by `meta_label_composite` (and regime multiplier) before the MAX_POSITION_WEIGHT clamp — via `sizing.position_sizer.size_position()` (see below).
- Any code that unpacks `aggregate()` must now unpack 6 elements, not 5.
  - **sizing/kelly.py** — `estimate_win_rate_and_payoff(closed_trades_df, lookback_trades=100) -> (p, b, n_trades)`: estimates win probability and payoff ratio from the most recent closed trades (by `exit_ts`) in `transactions_store.TransactionsStore.closed_trades_df()`. Returns `(NaN, NaN, n)` if `n_trades < 30` (`MIN_TRADES_REQUIRED`) or if there are no losing trades in the sample (b undefined) — logs an error below 30, a warning below 50 (`MIN_TRADES_FOR_CONFIDENCE`). `fractional_kelly(p, b, fraction=0.5, cap=0.20) -> float`: `f* = (p*b - (1-p))/b`, scaled by `fraction` and capped at `cap`; returns NaN if `p`/`b` is NaN so callers can detect "insufficient history" and fall back.
  - **sizing/vol_target.py** — `volatility_target_weight(realized_vol, target_vol=0.10, max_leverage=2.0) -> float`: `target_vol/realized_vol`, capped at `max_leverage`, saturating at the cap (not dividing by zero) when `realized_vol <= 0`. `portfolio_vol_target(positions, cov_matrix, target_vol, max_leverage) -> dict[symbol, weight]`: scales a raw position vector by one scalar so `sqrt(w^T Σ w) == target_vol` (capped); symbols missing from `cov_matrix` are excluded and explicitly set to `0.0` (logged), never fabricated.
  - `StrategyEngine.__init__` now accepts an optional `transactions_store` (DI for tests, e.g. `TransactionsStore(db_url="sqlite:///:memory:")`); defaults to lazily constructing a real `TransactionsStore()`. `_calculate_kelly_sizing(realized_vol)` is the single source-of-truth sizing call: if `estimate_win_rate_and_payoff` returns real `(p, b)`, sizing = `fractional_kelly(p, b, fraction=settings.KELLY_FRACTION, cap=settings.KELLY_CAP)`; otherwise (insufficient trade history) it logs a warning and falls back to `volatility_target_weight(realized_vol, target_vol=settings.VOL_TARGET, max_leverage=settings.MAX_LEVERAGE)` — **no Kelly multiplier** in the fallback. Either path is then clamped to `settings.MAX_POSITION_WEIGHT` (1.0) via `_raw_kelly_or_vol_target_sizing()` — a hard single-name ceiling independent of sizing methodology, since the vol-target fallback alone can otherwise reach `MAX_LEVERAGE` (2.0x). `main_orchestrator.py` now reads `strategy_output['Kelly Target']` directly (the duplicate `ee.calculate_kelly_target(...)` call + score-bracket override at that call site is gone). `realized_vol` is sourced from `garch_vol` (the per-ticker GJR-GARCH volatility already flowing into `evaluate_security`).
  - **Live data note:** `quant_platform.db` is per-machine runtime state (gitignored via `*.db`, not checked into git — see the "Historical/runtime data is never committed" convention below); a fresh checkout starts with an empty `trades` table (0 rows) via `TransactionsStore`'s own `Base.metadata.create_all()`. The closed-trade population that feeds `_calculate_kelly_sizing()` is **reconstructed on demand** (PURE FIFO lot-matching of a Robinhood account's filled equity order history) by `data/robinhood_orders.py` (Tier 7), and accumulates live as advisory runs call `record_trade()`. Until at least `MIN_TRADES_REQUIRED` (30) closed trades exist, `_calculate_kelly_sizing()` takes the **vol-target fallback path** (not the real Kelly path). Tests that assert on `Kelly Target` should still inject `TransactionsStore(db_url="sqlite:///:memory:")` explicitly for determinism (see `tests/test_quantitative_models.py::test_garch_and_edge_scoring` for the pattern) so they never depend on whatever rows happen to be in the on-disk DB.
  - **sizing/position_sizer.py (sizing-decision pipeline + portfolio cap + guardrail telemetry):** consolidates the previously-scattered regime/meta-label/`MAX_POSITION_WEIGHT` clamp composition into ONE ordered function, `size_position(pre_regime_weight, *, regime_multiplier, meta_label_composite, max_position_weight, path_tag="", raw_weight=None, kelly_cap=None, max_leverage=None, recent_cap_events=None, escalation_threshold=None, escalation_factor=None) -> SizingDecision`. Does NOT reimplement the Kelly/vol-target math (still `sizing/kelly.py` / `sizing/vol_target.py`) — it composes their already-computed outputs and reports which hard ceiling (if any) bound this cycle: `SizingDecision.was_capped: bool` / `.binding_constraint: str|None` (`"kelly_cap"`, `"vol_target_leverage"`, `"max_position_weight"`, `"portfolio_gross"`, `"escalation"`, or `None`) — telemetry that did not exist anywhere in the codebase before this. **Deliberately excludes the routine HMM regime multiplier** from `was_capped`/`binding_constraint`: `regime_multiplier < 1.0` on a risk-off day is expected signal-driven derating, not a guardrail cap, and folding it in would fire the guardrail on almost every risk-off cycle and drown out genuine ceiling events in the audit log/alerting; `Regime_Multiplier` remains its own separately-surfaced float, unchanged. `StrategyEngine._calculate_kelly_sizing()` itself is UNCHANGED (its own `MAX_POSITION_WEIGHT` clamp on the raw per-symbol weight is a directly-tested contract — `tests/test_kelly_no_history.py`, `tests/test_kelly_order_sizing.py`, `Gravity AI Review Suite.py` step 16 all call it standalone and assert the clamp fires); a new `_calculate_kelly_sizing_detailed()` returns `(raw_weight, clamped_weight, path_tag)` so `evaluate_security()` can feed BOTH values into `size_position()` for full telemetry without computing the (potentially 1,000-resample bootstrap) sizing path twice. Numeric-parity proven exactly (not just tested): at every existing scenario the refactored pipeline's `final_weight` is byte-identical (`0.00e+00` diff) to the pre-refactor inline arithmetic — confirmed both by the full pre-existing test suite passing UNCHANGED with its original expected values, and by a direct scenario-by-scenario equivalence script (see "Behavior-change validation" below).
    - **Portfolio-level gross exposure cap** (new constraint, layered on top of — not instead of — the per-name `MAX_POSITION_WEIGHT` ceiling): `apply_portfolio_gross_cap(per_name_weights, *, max_gross, cov_matrix=None, target_vol=None) -> PortfolioCapResult`. Reuses (does not reimplement) `sizing.vol_target.portfolio_vol_target` when a covariance matrix is supplied (risk-aware `target_vol/sqrt(w^T Σ w)` path); otherwise falls back to a simple sum-of-`|weight|` gross-exposure scalar. Both paths apply exactly ONE uniform scalar across every name (preserves relative sizing). Applied by `pipeline/production_steps.py::StrategyEvalStep` as a cycle-wide post-pass AFTER every name's own sizing (including the Dual Momentum overlay), against `settings.MAX_PORTFOLIO_GROSS` (default `3.0` — non-binding for a typical diversified book at `MAX_POSITION_WEIGHT=1.0` each; binds for a 10+ name book where every name is independently maxed out — see the sensitivity sweep below). A name the cap reduces has its `Sizing_Was_Capped`/`Sizing_Binding_Constraint` overridden to `"portfolio_gross"` (applied chronologically last, the most authoritative reason a position ended up smaller than its raw recommendation).
    - **Guardrail telemetry surfaces:** `"Sizing_Was_Capped"` / `"Sizing_Binding_Constraint"` are new `config.COLUMN_SCHEMA` entries (Sheet column "Sizing Capped" / "Sizing Binding Constraint", right after "Kelly Size") — schema-driven, so `get_headers()`/`get_internal_keys()` carry them to Sheets automatically. Also surfaced in `output/state_snapshot.json` per-signal dict (`sizing_was_capped`/`sizing_binding_constraint`, both writers — `main_orchestrator._write_state_snapshot` AND the advisory `reporting/state_snapshot.write_state_snapshot`, enforced by `tests/test_state_snapshot_parity.py`'s `SHARED_SIGNAL_FIELDS` living contract) and the GUI (Strategy Matrix's regime-impact section; Observability's new "Sizing Cap-Event Audit Trail" section). **Advisory path (`engine/advisory.py`) is deliberately NOT routed through `size_position()`'s full pipeline** (no regime/meta-label composition to do) — its own `CONFIG["kelly_cap"]`/`CONFIG["max_single_position_pct"]` (a tighter 5% single-name ceiling, intentionally decoupled from `settings.MAX_POSITION_WEIGHT` — see CONFIG's "Advisory-layer position size cap" note) gets its OWN guardrail telemetry via `_compute_kelly_sizing_detailed()`, surfaced as `Recommendation.sizing_was_capped`/`.sizing_binding_constraint`. It DOES reuse `sizing.position_sizer`'s two public comparison helpers instead of re-deriving them (CONSTRAINT #7): `detect_raw_cap_binding(path_tag, raw_weight, kelly_cap, max_leverage)` (the same raw-formula-cap detector `size_position()`'s own step 1 uses) and `clamp_with_binding(value, ceiling, constraint_name)` (clamp-and-report, used by both `size_position()`'s step 3 and advisory's own single clamp stage) — plus `KELLY_CAP`/`VOL_TARGET_LEVERAGE` string constants for a consistent vocabulary, plus its own `ADVISORY_MAX_POSITION_PCT` constraint name since its ceiling is a different, tighter setting.

**Dead-lettered ticker telemetry (CONSTRAINT #4):** a ticker whose strategy evaluation raised this cycle (see `output/dead_letter.json`) must NOT surface `Sizing_Was_Capped`/`Sizing_Binding_Constraint` as a fabricated "not capped" — `pipeline/production_steps.py` defaults these two (unlike the generic string `""` default every other schema-driven `strategy_col` gets) to `None` when the ticker is absent from `eval_results`, and `main_orchestrator.py`'s `_safe_bool_or_none()` helper preserves that `None` through to `state_snapshot.json` as JSON `null` rather than collapsing a missing/NaN value into an active `false` claim (a plain `str(x).lower() == "yes"` coercion would do exactly that). The cap-event audit writer (`sizing/cap_audit_store.py` call site) likewise skips a dead-lettered ticker entirely rather than persisting a fabricated `was_capped=False` row, which would otherwise corrupt the cap-aware escalation rule's consecutive-capped-cycles read for that symbol. Covered by `tests/test_state_snapshot_parity.py::TestSizingGuardrailNullSafety` and `tests/test_main_orchestrator.py::TestSafeBoolOrNone`.
    - **Cap-aware escalation** (opt-in, `settings.SIZING_CAP_ESCALATION_ENABLED`, default `False` — zero behavior change until an operator opts in): a name that binds a hard ceiling for `>= settings.SIZING_CAP_ESCALATION_THRESHOLD_CYCLES` (default 5) consecutive cycles is further down-weighted by `settings.SIZING_CAP_ESCALATION_FACTOR` (default 0.5). Reads consecutive-capped-cycle history from `sizing/cap_audit_store.py`, ONLY when the flag is on (a disabled operator sees zero added DB reads). `StrategyEngine` gained a lazily-constructed `cap_audit_store` property (mirrors the existing `transactions_store` property's degrade-to-offline-stub contract on a DB outage — an audit-log failure must never itself trigger escalation nor abort sizing for a symbol).
  - **sizing/cap_audit_store.py** — durable SQLAlchemy log of every capping event (`sizing_cap_events` table: timestamp, cycle_id, symbol, strategy_id, raw_weight, final_weight, binding_constraint, was_capped), resolved through `db_config.py` and following `desktop/run_history_store.py`'s exact convention (own `Base`, a `readonly=True` database-level read-only engine, write methods raise on failure — CONSTRAINT #4 — read methods degrade to `[]`/a neutral `CapEventSummary(0)` on any error — CONSTRAINT #6). `get_consecutive_capped_cycles(symbol, strategy_id=None)` builds the `CapEventSummary` the escalation rule above consumes by walking the most-recent events backwards and stopping at the first non-capped one (a genuinely consecutive run, not a raw count over a window). Gated by `settings.SIZING_CAP_AUDIT_ENABLED` (default `True`, matching the `HISTORICAL_STORE_ENABLED` precedent for new persistence-layer infrastructure); a dead-lettered ticker (see the "Dead-lettered ticker telemetry" note above) is skipped entirely rather than persisting a fabricated event. `pipeline/production_steps.py::StrategyEvalStep` writes the whole cycle's events in ONE transaction after the portfolio cap above, and — only when `settings.SIZING_CAP_ALERT_ENABLED` is `True` (default `False`; KELLY_CAP binding is a routine, expected event for an established Kelly book, so this alert is opt-in rather than surprising every existing deployment with new console/file WARNING logs on upgrade) — fires a `observability.alerts.send_alert("WARNING", ...)` (the platform's unified multi-channel dispatcher — console/file always-on plus discord/slack/email when configured; NOT the separate legacy `ALERT_WEBHOOK_URL` POST, which is `execution/order_manager.py`'s reconciliation-drift-specific mechanism) when `>= settings.SIZING_CAP_ALERT_THRESHOLD_PCT` (default 30%) of this cycle's names were capped. Both the audit write and the alert are best-effort (a failure only logs a warning, never affects the run's own sizing decisions or SUCCEEDED/FAILED state).

Proof, run offline (this repo's sandboxed CI/dev environments have no outbound network access to yfinance/FRED, so a live-market PBO/DSR/Sharpe/MaxDD re-verification via `validation/harness.py` could not be executed as part of this change — see the introducing PR description for the honest bottom line and the recommended follow-up before deploying non-default `MAX_PORTFOLIO_GROSS`/escalation settings live): (1) a direct scenario-by-scenario equivalence check confirms `size_position()`'s `final_weight` is **byte-identical** (`0.00e+00` diff) to the pre-refactor inline formula across representative regime-multiplier/meta-label/ceiling scenarios; (2) a portfolio-gross-cap sensitivity sweep confirms the new cap does NOT bind for realistic diversified books (5-10 names at typical Kelly weights) at the shipped `MAX_PORTFOLIO_GROSS=3.0` default, and DOES bind — as intended — once many names are simultaneously sized at the full `MAX_POSITION_WEIGHT` ceiling (e.g. 10 names × 1.0 → gross 10.0 → scaled to 30%); (3) cap-aware escalation is confirmed a true no-op when `SIZING_CAP_ESCALATION_ENABLED=False` (the shipped default).
