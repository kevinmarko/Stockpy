# Architecture: Webapp, Desktop Shell & GUI (Command Center)

> Part of the split-out `CLAUDE.md` System Architecture reference. See [../../CLAUDE.md](../../CLAUDE.md) for the index.

- **webapp/ (Pilots PWA)** — mobile-first installable PWA (Vite + React + TypeScript + react-router + Recharts + vite-plugin-pwa) modeling the Autopilot consumer UX over `api/pilots_api.py`. Dark fintech theme (green `#10b981`/red `#ef4444`/amber `#f59e0b` on `#0b0e11`/`#12161c`, reusing `gui/styling.py`'s palette). Screens (1:1 with Autopilot): 3-step **Onboarding** (Choose Pilot → Connect brokerage paper-first → Set amount; completion in localStorage — the brokerage step offers Paper trading / **Connect Robinhood** / Browse-only-for-now; the Robinhood option (2026-07, new; switched from a persisted TOTP secret to a one-time code 2026-07) reveals inline username/password/one-time-6-digit-authenticator-code fields and calls `api.connectBrokerage(...)` (`POST /brokerage/connect`), showing verifying→connected/error state and blocking `Continue` until a real verified connection or a different option is chosen — the submitted password and code are never displayed once connected and the form unmounts on success; `onboarding.ts`'s `OnboardingState.brokerage` union gained `"robinhood"`), **Dashboard** (`/`: draggable widget grid — the app's home route), **Marketplace** (`/marketplace`: Top-Performers + Most-Popular rails + category groupings), **Pilot Detail** (`/pilots/:id`: perf chart with 1W–2Y toggles, holdings, sector-allocation **donut**, recent-trades feed, deployable/PBO/DSR honesty badges, Follow CTA), **Portfolio** (`/portfolio`: equity tiles + equity-curve + active follows), the **Follow modal** (amount input, `planned_intents` preview, execution-mode badge + an unmissable "this creates a gated queue you must confirm — no order is placed automatically" notice), and **Settings** (`/settings`: 2026-07, new — the "Data & Automation" dashboard over `api/pilots_api.py`'s `GET/POST/PUT /automation/*` endpoints; see that module's CLAUDE.md entry above for the full read/write contract. Pipeline status with a live "Run now" trigger, a "Signal generation: Running/Paused" `Toggle` for pause/resume — each direction confirmed via `Modal` + a required reason `Input`, resume disabled from the paused state when `advisory_only` is false — a schedule view with an opt-in interval `Input`+Save, per-pilot "Re-plan" buttons reusing the existing `POST /pilots/{id}/follow`, and the folded-in `PwaStatusSection` (formerly a standalone `PwaStatusDrawer` bottom-sheet — the fixed ⚙ gear on every screen now navigates to `/settings` instead of opening a local sheet, keeping the `needRefresh` update dot). Polls `/automation/status` every 3s ONLY while a run is actually in flight (`usePoll`, `src/hooks/usePoll.ts`) — never a phone's radio budget spent polling a status that changes every few minutes. Every write uses `useMutation` (`src/hooks/useMutation.ts`) and renders whatever the server actually returned, never assumes success client-side.). **Runs fully offline against a mock API layer** (`src/api/mock.ts`); `src/api/client.ts` flips mock→live via `VITE_USE_MOCK=false` (+ `VITE_API_BASE_URL`/`VITE_API_TOKEN`), with `src/api/types.ts` the single schema-reconciliation point mirroring the Phase-2 response shapes. **UI honesty (CONSTRAINT #4):** a non-deployable pilot renders `deployable=false` plainly and a `curve:null` renders "no backtest series yet", never a fabricated line. `node_modules`/`dist` gitignored; the `*.json` repo-wide ignore means `package.json`/`tsconfig.json`/`package-lock.json` are force-added.
- **gui/daemon_client.py** — thin, dependency-free (stdlib `urllib.request`/`json` only — this codebase's established convention for small internal HTTP calls, matching `desktop/net_util.py`'s `wait_for_http` and `alerting.py`'s webhook POSTs, despite `requests` being available for heavier external API work) HTTP client for `api/control_api.py`, used by `gui/orchestrator_runner.py`'s daemon fast path. Every function is non-raising (CONSTRAINT #6) — connection failures, timeouts, and malformed responses all degrade to a documented sentinel (`None`/`False`/a result dataclass), never an exception; failures log at DEBUG (a down/disabled daemon during normal operation is routine, not alarming). `daemon_available(timeout=0.5) -> bool`; `get_status(timeout=2.0) -> Optional[dict]`; `trigger_run(timeout=5.0) -> TriggerResponse` (frozen dataclass: `ok`, `run_id`, `state`, `error` — a stable code-like tag: `"already_running"`, `"kill_switch_active"`, `"unauthorized"`, `"command_disabled"`, `"unavailable"`, `"network_error"`, `"unexpected_response"` — plus `existing_run_id`/`kill_switch_reason` populated only for their respective cases); `get_run_status(run_id, timeout=2.0) -> Optional[dict]`; `get_latest_run(timeout=2.0) -> Optional[dict]`; **`set_interval(interval_seconds, timeout=5.0) -> IntervalResponse`** (2026-07, live interval setter — the first PUT/body-sending function in this module; `IntervalResponse` frozen dataclass: `ok`, `interval_seconds`, `error` — same stable-tag contract as `TriggerResponse`, plus a `"invalid_interval"` tag for a 422 the Control API's pydantic validator rejected before it ever reached the daemon). Base URL and the `Authorization: Bearer <token>` header (omitted entirely, not sent empty, when `settings.ORCHESTRATOR_DAEMON_TOKEN` is unset) are both read from `settings` live at CALL time, not cached, so a settings change takes effect without reimporting the module. A leaf module — no imports from `desktop.daemon_runtime`, `api.control_api`, `gui.orchestrator_runner`, or any pipeline/engine module.
- **app_shell.py** / **desktop/** — Native desktop shell for the unified Command Center. `app_shell.py` opens `gui/app.py` in a real OS window (pywebview) instead of a browser tab, and owns an always-on background refresh loop tied to the window's lifecycle (starts when the window opens, stops when it closes). `desktop/` holds the shell's supporting modules: `net_util.py` (local port/URL helpers for the embedded server), `ui_server.py` (runs the Streamlit app as a local server the native window points at), and `engine_supervisor.py` (starts/stops/monitors the background refresh loop and exposes its freshness for the sidebar indicator). **`engine_supervisor.start_engine`/`stop_engine` signatures are frozen** — `app_shell.py` depends on them unchanged — but the BODY of `start_engine` now branches on `settings.ORCHESTRATOR_DAEMON_ENABLED` (default `False`): off (default) spawns `main.py --interval N` via `gui.orchestrator_runner.launch_scheduled_advisory` exactly as before; on spawns `python -m desktop.orchestrator_daemon --interval N` via `gui.orchestrator_runner.launch_daemon_engine` instead — still an ordinary supervised `Popen` child (`backend="subprocess"`) using the identical `RunHandle`/`stop_run` SIGTERM→SIGKILL mechanics, so `stop_engine` needed zero changes. Exactly one of the two loops is ever spawned, never both.
  - **desktop/daemon_runtime.py** — `OrchestratorDaemon`: the persistent orchestrator daemon's signal-agnostic, thread-safe, single-flight run engine (the class that actually keeps engines warm across cycles). `start()` builds one warm `DataEngine` + `EngineContext` (mirroring `_main_body`'s own `credentials.json`-based DataEngine/MockDataEngine choice, paying the construction cost once); idempotent (a second call logs a warning and no-ops). `trigger_run(*, reason="manual"|"interval") -> TriggerResult` is non-blocking and single-flight: a run already in progress returns `TriggerOutcome.ALREADY_RUNNING` + the existing run's id (never a second concurrent cycle, never a queue). Each accepted trigger spawns a worker thread that drives `main_orchestrator._main_body(dry_run, strict=strict, engines=self._engines, data_engine=self._data_engine)` directly — NOT `main()`'s asyncio-heartbeat wrapper, which manages its own per-call dotenv/heartbeat lifecycle unsuited to a long-lived process. Both `PipelineFatalError` and any unexpected `Exception` are caught and recorded as a `RunState.FAILED` `RunRecord` — **the daemon survives a crashed cycle instead of dying with it**, the entire point of this redesign. `RunRecord` (frozen dataclass: `run_id, state, started_at, finished_at, duration_seconds, error, reason, progress`) is inserted as a `RunState.RUNNING` placeholder at trigger time (same lock acquisition that claims the single-flight slot) and overwritten in place on completion, so `get_run(run_id)` finds an in-flight run immediately rather than only after it finishes. `progress` mirrors `reporting.progress.read_progress()`'s shape for the in-flight/most-recent run so `api/control_api.py` and `gui/daemon_client.py` can surface live pipeline progress without the caller reading `output/progress.json` itself. Bounded run history (`run_history_size`, default 10; oldest evicted first) via `get_run()`/`status()`/`last_result`. **Durable persistence (2026-07, new):** immediately after building each completed `RunRecord`, `_run_one_cycle` ALSO best-effort persists it to `desktop/run_history_store.py`'s durable `pipeline_runs` DB table (lazy import, matching `HistoricalStore`'s convention elsewhere) — a DB write failure only logs a warning and never affects the run's own SUCCEEDED/FAILED verdict, exactly like the `progress_snapshot` capture just above it. See that module's own entry below. **Live interval setter (2026-07):** `set_interval(interval_seconds)` (validated via `settings.validate_interval_seconds`) changes the timer cadence WITHOUT a restart, creating the timer thread on demand if none exists (`start()` only creates one when `interval_seconds > 0` at startup, so a daemon started at the default 0 — on-demand only — has no thread to signal otherwise). The timer loop uses TWO `threading.Event`s (not a `Condition` — zero precedent for that primitive in this codebase): `_stop_event` (set once, at shutdown) and `_wake_event` (cleared and set repeatedly, by `set_interval()` to wake a parked/waiting loop immediately and by `shutdown()` so a PARKED loop — `interval_seconds <= 0`, blocked on an untimed `_wake_event.wait()` — actually wakes; `_stop_event` alone would never reach it). `_timer_loop` clears `_wake_event` BEFORE reading `_interval_seconds` each iteration — if `set_interval()` fires in that exact window, the loop still observes the NEW value on the same iteration (one harmless spurious pass), never sleeps out a stale one. `shutdown(timeout=10.0)` sets BOTH events and reads/clears the thread reference under the lock but joins OUTSIDE it (`_timer_loop` may call `trigger_run()`, which itself acquires the same lock — holding it across `join()` would deadlock against a timer thread mid-`trigger_run()`). Owns NO `signal`/SIGTERM/process-lifecycle handling at all — that is `desktop/orchestrator_daemon.py`'s separate concern, by design.
  - **desktop/run_history_store.py** — `RunHistoryStore`: SQLAlchemy-backed durable log of every `OrchestratorDaemon` run cycle, in its own `pipeline_runs` table (own `Base`, resolved through `db_config.py` exactly like `transactions_store.py` — SQLite by default, Postgres/Supabase when `DATABASE_URL` is set). Exists because `OrchestratorDaemon._run_history` (above) is a plain in-memory dict capped at `run_history_size` and lost on every daemon restart — the Pipeline Dashboard's run-history table had nothing once the daemon restarted. `record_run(record)` upserts by `run_id` (write methods intentionally still raise — CONSTRAINT #4 — the caller, `_run_one_cycle`, wraps the call in its own best-effort try/except). `readonly=True` builds a database-level read-only engine (mirrors `TransactionsStore`, skips `Base.metadata.create_all`) for `api/control_api.py`'s `GET /runs/history` (see `docs/architecture/observability-and-apis.md`); `get_recent(limit)` degrades to `[]` on any read failure (CONSTRAINT #6), never raises. Only terminal (succeeded/failed) runs are ever written — a run still in flight only ever appears in the in-memory ring above, never this table, by design. The Pipeline Dashboard's "Full run history" section (`webapp/src/screens/PipelineDashboard.tsx`, distinct from the same screen's existing "Run history" table which still reflects the in-memory ring via `GET /status`) polls this on-demand — a manual Refresh button plus an auto-refresh the moment an in-flight run completes — rather than the 3s live poll the in-memory table uses. Tests: `tests/test_run_history_store.py`.
  - **desktop/orchestrator_daemon.py** — standalone process entrypoint: `python -m desktop.orchestrator_daemon [--interval N] [--dry-run] [--strict]`. Constructs and starts an `OrchestratorDaemon`, hosts `api/control_api.py`'s FastAPI app via `uvicorn.Server` on a background thread (`127.0.0.1:settings.ORCHESTRATOR_API_PORT`, default 8601 — loopback-only, defense-in-depth alongside the Control API's auth tokens), then writes `output/daemon.json` (atomic write-then-rename, mirrors `execution/kill_switch.py`'s `activate()` idiom) with `{pid, state, interval_seconds, started_at, port, pilots_api_port}` for external discovery — **only after** a bounded (5 s) poll confirms the Control API (and, when enabled, the Pilots API) actually started listening (`uvicorn.Server.started`), so a discovery file is never written pointing at a not-yet-bound port; falls through and writes it anyway past the deadline (logged) rather than blocking startup indefinitely. **Optional second service — the Pilots API (`api/pilots_api.py`, `settings.PILOTS_API_ENABLED`, default `False`):** when `True`, `run_forever()` also constructs a second `uvicorn.Server` for `api.pilots_api:app` on `127.0.0.1:settings.PILOTS_API_PORT` (default 8602, matching the standalone-launch port), started via its own background thread (`name="PilotsAPI"`) right after the Control API thread, awaited by the same bounded readiness poll, and torn down (`should_exit=True` + `join(timeout=5.0)`) alongside the Control API in `_teardown()`. A construction/import failure for this OPTIONAL service is caught, logged, and swallowed — it can never abort daemon startup or affect the always-on Control API (CONSTRAINT #6); `pilots_api_port` in the discovery file stays `None` when disabled or on that failure path. `False` (the default) reproduces every pre-existing behavior byte-for-byte — `api/pilots_api.py` remains a manually-launched standalone `uvicorn` process unless explicitly opted in. Tests: `tests/test_orchestrator_daemon.py::TestPilotsAPIHosting`. **New env vars:** `PILOTS_API_ENABLED` (bool, default `False`, non-secret, GUI-writable), `PILOTS_API_PORT` (int, default 8602, `.env`-only like `ORCHESTRATOR_API_PORT`). **SIGTERM/SIGINT hardening reuses `app_shell.py`'s exact proven pattern** (`signal.pthread_sigmask(SIG_BLOCK, ...)` + a dedicated thread blocking on the genuinely-blocking `signal.sigwait()` + idempotent teardown + `os._exit(0)`) — a plain `signal.signal()` handler has a documented, previously-shipped bug in this codebase (PRs #160/#163) where it silently failed to terminate a real running process. **Critical ordering constraint, found via live-process verification (not caught by any mocked test):** `pthread_sigmask(SIG_BLOCK, ...)` MUST be the very first thing `run_forever()` does, before `daemon.start()` (which may spawn the interval-timer thread) or the Control API's `uvicorn` thread are created — `pthread_sigmask` sets the CALLING THREAD's mask, not a process-wide one, so a thread created before the mask is blocked inherits an UNBLOCKED mask for its own lifetime and stays eligible for the kernel's default SIGTERM disposition; a real `kill -TERM` could then be delivered straight to that thread instead of the `sigwait` watcher, silently killing the whole process with ZERO log output and no teardown at all. Regression-guarded by `tests/test_orchestrator_daemon.py::TestSignalHandling::test_signals_blocked_before_any_thread_or_daemon_start` (asserts call ordering; verified to fail against the pre-fix ordering).
- **gui/** — **InvestYo Command Center**: a local-first, on-demand Streamlit operational suite over the full quant lifecycle, now primarily launched inside the native desktop shell (`./launch_app.command` → `app_shell.py`) but still runnable standalone via `streamlit run gui/app.py` or `./launch_gui.command`. Read-only / file-backed by design — it **never calls async broker code directly**; it launches `main_orchestrator.py` as a **subprocess** and consumes the file-backed state the orchestrator writes (`output/state_snapshot.json`, `output/heartbeat.txt`, `output/KILL_SWITCH`). Modules:
  - **gui/app.py** — entry point. Repo-root `sys.path` shim + `load_dotenv(override=False)`, `st.set_page_config`, sidebar, and 18 `st.tabs` (the `tab_labels` list; the 18th, index 17, is `"🔬 Validation Lab"`). Every tab body is wrapped by `safe_panel()` so one panel's exception renders an inline error box instead of crashing the app (dead-letter UI; CONSTRAINT #6).
  - **gui/env_io.py** — safe, allowlist-bounded `.env` read/write for the Settings & Strategy-Matrix tabs. `ALLOWED_KEYS` (NON-secret tunables only) are writable; `SECRET_KEYS` (API keys, passwords, TOTP, webhooks) are **masked** by `read_settings()`/`mask_secret()` and raise `SecretWriteError` if a write is attempted (CONSTRAINT #3). Non-allowlisted keys raise `DisallowedKeyError`. List/dict tunables (`DEFAULT_TICKERS`, `SIGNAL_WEIGHTS`, `DISABLED_SIGNAL_MODULES`) are JSON-encoded so pydantic-settings re-parses them. `MACRO_REGIME_GATE_ENABLED` is in the Risk gate section of `ALLOWED_KEYS`. Uses `dotenv.dotenv_values` + `dotenv.set_key` (preserves comments/other lines). Writes take effect on the **next** launch (no hot-reload).
  - **gui/orchestrator_runner.py** — `launch_orchestrator(dry_run, refresh_account) -> RunHandle` spawns `[sys.executable, "main_orchestrator.py", ...]` via `subprocess.Popen` (non-blocking, stdout→`output/gui_run.log`). `compute_stage_status()` derives coarse Data Acquisition/Processing/Forecasting/Execution stage status from log markers + `heartbeat.txt` freshness + `state_snapshot.json` mtime. `read_log_tail()` / `heartbeat_age_seconds()` feed the Launcher tab. **Persistent-daemon cutover (`settings.ORCHESTRATOR_DAEMON_ENABLED`, default `False`):** `launch_orchestrator()` gains a flag-gated fast path — when the flag is on, a daemon is reachable (`gui.daemon_client.daemon_available()`), and `dry_run=False` (an explicit dry-run always falls back to a real subprocess, since an already-running daemon has a FIXED `dry_run` set at process launch with no per-trigger override), it triggers a cycle over HTTP via `gui.daemon_client.trigger_run()` instead of spawning a new process, returning a `backend="daemon"` `RunHandle`. Falls back to the unchanged subprocess path on ANY failure (daemon down, trigger rejected, unexpected exception — CONSTRAINT #6, this fast path can never block a manual run). `launch_daemon_engine(interval_seconds, refresh_account) -> RunHandle` is the separate always-on-loop launcher `desktop/engine_supervisor.py`'s `start_engine` calls when the flag is on — spawns `python -m desktop.orchestrator_daemon --interval N`, reusing the EXACT SAME `Popen`/`RunHandle`/`stop_run` mechanics as `launch_scheduled_advisory` (`backend="subprocess"`, no dual-backend logic needed); `refresh_account` is accepted for call-site symmetry but silently ignored (the daemon entrypoint has no `--refresh-account` equivalent). `RunHandle` gained `backend` (`"subprocess"` default | `"daemon"`) and `daemon_run_id` fields; `is_running()`/`returncode()` branch on `backend`, polling `gui.daemon_client` for a daemon-hosted run instead of a local `Popen` (`returncode()` synthesizes `0`=succeeded/`1`=failed/`None`=queued-or-running from the Control API's run state). `stop_run()` on a `backend="daemon"` handle logs a clear "cannot cancel" warning and returns `False` — the Control API exposes no cancel/abort endpoint, so an in-flight daemon-hosted cycle always runs to completion; it does NOT pretend to support cancellation. `read_log_tail()` synthesizes a short status summary (state/timestamps/error) for a daemon-hosted run instead of reading a nonexistent per-run log file. `compute_stage_status()` maps the coarse daemon run state onto a uniform per-stage status (`succeeded`→all `SUCCESS`, `failed`→all `ERROR`, `running`/`queued`→all `ACTIVE`) since there's no local log to scan for stage markers. **Validation Lab launcher:** `launch_validation_run(strategies, start, end) -> RunHandle` spawns `[sys.executable, "-m", "scripts.refresh_validations", "--strategies", …, "--start", …, "--end", …]` as a non-blocking `subprocess.Popen` child (mirroring the existing `launch_pytest` pattern — same `RunHandle`/`stop_run` mechanics), streaming stdout to `VALIDATION_LOG_PATH = settings.OUTPUT_DIR / "gui_validation.log"`. On-demand only (no scheduling); the report artifacts land in `reports/` + `reports/history/` unchanged. **Live progress:** `compute_run_progress(handle) -> Optional[RunProgress]` (`RunProgress`: `percent`, `label`, `indeterminate`) reads `reporting.progress.read_progress()` for a subprocess-backed run (or the daemon's `RunRecord.progress` for a `backend="daemon"` handle) and normalizes both into one shape for the Launcher tab's live `st.progress` bar (polling at `settings.PROGRESS_POLL_SECONDS`); `None` when no run is active or no progress data exists yet (never fabricated — CONSTRAINT #4).
  - **gui/progress_ui.py** — `busy(label)` is a lightweight busy-indicator context manager, independent of `reporting/progress.py`'s stage/symbol-level `ProgressReporter` — meant for previously-silent one-shot GUI actions (a button click that kicks off a synchronous operation with no discrete stage structure to report, e.g. "Sync Now" on Live Inventory or "Run" on the Validation Lab) rather than the multi-stage pipeline run. Wraps the action in a Streamlit spinner/status element labeled `label` so every long-running click gives immediate visual feedback instead of an unresponsive-looking UI.

Paper-Trading Monitor (`data.robinhood_portfolio.fetch_account_snapshot()` account truth vs. pipeline projection, source-labeled per CONSTRAINT #4); Gravity Audit Logs (runs `Gravity AI Review Suite.py` as a subprocess, parses trailing JSON, shows pass/fail);
