# Architecture: Simulation, Evaluation & Reporting Plumbing

> Part of the split-out `CLAUDE.md` System Architecture reference. See [../../CLAUDE.md](../../CLAUDE.md) for the index.

- **simulation_engine.py** — `vectorbt` parameter-sweep optimization plus `backtrader` event-driven backtests (0.1% commission / 0.05% slippage standard). Integrates survivorship bias warning prints using `universe_engine.py`. New strategies should be proven here before being wired into `strategy_engine.py`.
- **universe_engine.py** — S&P 500 point-in-time universe loader. Scrapes Wikipedia current and changes tables to reconstruct index constituents, loads local delistings from `data/delisted_tickers.csv`, and reports estimated survivorship bias.
- **research_engine.py** — realized slippage, CoVaR tail-dependency proxy, Brinson-Fachler attribution, MFE/MAE, portfolio heat.
- **evaluation_engine.py** — strategy performance evaluation. **`calculate_equity_curve_metrics(equity_df, risk_free_rate=0.0) -> Dict[str, float]`** (module-level, alongside `calibration_curve`) computes Sharpe ratio, Calmar ratio, max drawdown, max-drawdown duration (days), and CAGR from an account equity curve (the exact shape returned by `data.historical_store.HistoricalStore.account_snapshot_history()` — `fetched_at`/`total_equity` columns). Vectorized (dedupes multiple same-day snapshots to the last one via `groupby`/`drop_duplicates`, no `.iterrows()`); every value is `NaN` — never a fabricated `0.0` — when there are fewer than `MIN_SNAPSHOTS_FOR_STATS` (20) distinct daily snapshots or `total_equity` has zero variance (Sharpe/Calmar undefined for a flat curve). Surfaced as four additional `st.metric` tiles in the GUI Analytics tab's Account Value Equity Curve section (`gui/panels/analytics.py::_render_account_equity_curve`).
- **transactions_store.py** — SQLAlchemy-backed transactions database for tracking active/closed trades. Backend is resolved through **db_config.py** (dual-backend config seam): defaults to a local SQLite file (`quant_platform.db`, WAL journaling); setting `DATABASE_URL` (e.g. a `postgresql://...` DSN, including a Supabase Postgres project) transparently switches every store onto Postgres with a pre-pinged pool (`DB_POOL_SIZE`=5, `DB_MAX_OVERFLOW`=10). `db_config.create_db_engine()` never logs the full URL (CONSTRAINT #3 — a Postgres DSN embeds `user:pass@host`; only the backend name is logged). **DB-outage resilience (2026-07):** `TransactionsStore()` construction does an eager connection (`Base.metadata.create_all`), so a remote-backend connectivity failure (DNS/network — e.g. an unreachable Supabase host) used to raise straight out of `engine.advisory._get_transactions_store()` and `StrategyEngine.transactions_store`, dead-lettering EVERY symbol's advisory evaluation for the cycle over what is, for the advisory pipeline, an optional position-sizing refinement. Both of those lazy-construction call sites now catch a construction failure, log ONCE, and substitute **`transactions_store._OfflineTransactionsStore`** — a read-only stub whose `closed_trades_df()`/`open_trades_df()`/`get_trade_history()` return empty DataFrames (the same "zero closed trades" cold-start shape `sizing/kelly.py` already falls back from to `volatility_target_weight` — CONSTRAINT #6) while `record_trade()`/`close_trade()` still raise (a trade that was never persisted must not be fabricated as recorded — CONSTRAINT #4). The stub is cached on the singleton/property so a DB outage doesn't retry-storm the failing host once per ticker in the universe. Covered by `tests/test_transactions_store.py` and Gravity `step_75_db_backend_resilience_audit` (renumbered from `step_74` by PR #209's Gravity split-brain consolidation, which resolved a same-day collision with `step_74_progress_indicator_audit`). **Operational note:** Supabase's direct-connect hostname (`db.<project>.supabase.co`) is IPv6-only in many regions/projects; on an IPv4-only network this manifests as `psycopg2.OperationalError: could not translate host name ... to address` even though the DSN/credentials are correct — use Supabase's Session/Transaction pooler hostname (`aws-0-<region>.pooler.supabase.com`, username `postgres.<project-ref>`) instead, which is IPv4-compatible and free.
- **`.agents/skills/`** — Agent-context Skills surface for this repo. Contains two Skills: `.agents/skills/supabase/` and `.agents/skills/supabase-postgres-best-practices/`. Coding agents working the Postgres/Supabase backend of `transactions_store.py` / `db_config.py` (the `DATABASE_URL` dual-backend seam + Supabase pooler-hostname note above) should consult `.agents/skills/supabase-postgres-best-practices/` for connection-pooling, RLS, indexing, and transaction-locking guidance.
- **database_setup.py** — builds the SQLite schema (`DailySignals`, `ExecutionLogs`) dynamically from `config.COLUMN_SCHEMA`; never hardcode SQL column lists. `quant_platform.db` itself is **not** checked into git (gitignored via `*.db`, per-machine runtime state that accumulates local trades/signals/snapshots) — a fresh clone runs `python3 database_setup.py` once (creates `DailySignals`/`ExecutionLogs`), and `TransactionsStore`'s own `Base.metadata.create_all()` / `HistoricalStore`'s lazy table creation build the rest on first use. This was untracked from git on 2026-07-06 after a prior incident where `ml/registry.yaml` (a similar tracked, mutating file) was corrupted by a concurrent-PR merge — tracked files that mutate on every run are a recurring source of pull/merge data-loss risk.
- **diagnostics_and_visuals.py** — HTML report generation/rendering. `diagnostics_and_visuals.generate_html_report(portfolio_data, regime, output_path, *, yield_curve, credit_spread, sahm_rule, real_yield, audit_log=None, account_summary=None)` is the single HTML **rendering engine** (the legacy `reporting_engine.py`/`daily_report_template.html` pair was superseded and deleted 2026-07-09). It is no longer called inline by the orchestrators: `reporting/html_publisher.py` (`write_html_report`) is the extracted publishing layer that both `main.py` and `main_orchestrator.py` route through, and it in turn calls `generate_html_report` to render. **Report redesign (2026-06):** the embedded `HTML_REPORT_TEMPLATE` leads with **Holdings & P&L** (shares, avg cost, current price, market value, signed unrealized P&L $/%) and **Action & Rationale** (action signal w/ colour class + conviction meter, suggested position %, click-to-expand plain-English rationale). New optional keyword `account_summary: dict` renders a top **portfolio summary band** (total equity, buying power, total unrealized P&L, dividends received, position count, BUY/HOLD/SELL tally); when `None` (the `main_orchestrator.py` path) the band is hidden and rows degrade to "—" placeholders. Backward-compatible: field normalization accepts BOTH spaced pipeline keys (`"Action Signal"`, `"Robinhood Unrealized PL"`) and underscored advisory keys (`"Action_Signal"`, `"UnrealizedPL"`); market value / unrealized P&L are **derived** (`shares×price`, `(price−avgCost)×shares`) only when the Robinhood snapshot fields are absent (never fabricated for non-held symbols — those render "—"). Dependency-free client-side **search box + sortable columns** (vanilla JS, no DataTables/jQuery). `main.py._write_html_report()` sources holdings/P&L from the Robinhood `AccountSnapshot` positions (source of truth for account state, CONSTRAINT #4) and builds `account_summary` defensively (a degraded/empty snapshot omits the band, never aborts the report). Covered by `tests/test_html_report.py`.
- **reporting/html_publisher.py** — Report-publishing layer extracted out of `main.py` (`write_html_report(result, macro_dto)`, `write_state_snapshot`) so the Sheets + HTML publish logic is no longer inlined in the advisory entry point; both orchestrators call it instead. `write_html_report` renders via `diagnostics_and_visuals.generate_html_report` and also invokes `reporting.state_snapshot.write_state_snapshot` for the GUI-facing telemetry snapshot.
- **reporting/progress.py** — Cross-cutting, dead-letter-safe pipeline-progress reporter. `ProgressReporter(stages, *, run_id=None, output_dir=None)` is instantiated once per cycle by the orchestrator (`main_orchestrator.py`/`main.py`/`pipeline/*`) with the ordered list of stage names for that run; `start_stage(name, *, symbols_total=0)` advances to a new stage and resets the per-stage symbol counter; `advance_symbol(message="")` (thread-safe — safe to call from the parallelized per-ticker worker pools described under `main_orchestrator.py`/`engine/advisory.py` above) increments the completed-symbol count within the current stage; `set_message(msg)` updates a free-text status line without changing stage/counts; `finish(state="succeeded")` marks the run terminal (`"succeeded"` | `"failed"` | `"cancelled"`). Every call atomically writes `output/progress.json` (write-then-rename, mirroring `execution/kill_switch.py`'s `activate()` idiom) with the shape `{run_id, state, stage, stage_index, stage_total, symbols_done, symbols_total, percent, message, started_at, updated_at}`. Module-level `read_progress(output_dir=None) -> Optional[ProgressState]` is the read side — dead-letter resilient (CONSTRAINT #6): a missing, unreadable, or malformed `progress.json` degrades to `None`, never a fabricated/partial progress dict (CONSTRAINT #4). `clear_progress(output_dir=None)` removes the sentinel (e.g. at the start of a fresh run). Consumed by `desktop/daemon_runtime.py`'s `RunRecord` (gained a `progress` field mirroring the same shape, surfaced through `api/control_api.py`'s `GET /status` / `GET /run/{run_id}/status` and `gui/daemon_client.py`) and by `gui/orchestrator_runner.py::compute_run_progress(handle) -> Optional[RunProgress]` (`RunProgress` carries `percent`, `label`, `indeterminate` — `indeterminate=True` when a subprocess-backed run has no progress file yet, e.g. before the orchestrator's first `start_stage()` call). The Launcher tab (`gui/panels/launcher.py`) renders this as a live `st.progress` bar polling at `settings.PROGRESS_POLL_SECONDS` (default 5s, GUI-writable non-secret `ALLOWED_KEY`). **New GUI convention:** `gui/progress_ui.py::busy(label)` is a lightweight busy-indicator context manager (independent of `ProgressReporter` — for actions with no discrete stage/symbol structure, e.g. a single button click) applied across previously-silent GUI buttons so every long-running action gives immediate visual feedback.
