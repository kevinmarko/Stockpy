# Architecture: Execution (Broker, Risk Gate, Order Queue)

> Part of the split-out `CLAUDE.md` System Architecture reference. See [../../CLAUDE.md](../../CLAUDE.md) for the index.

- **execution/cost_model.py** — Tiered execution cost model and Backtrader CommissionInfo (SEC §31, FINRA TAF, tiered spreads, market order slippage).
- **execution/broker_base.py** — `BrokerBase` ABC: `OrderSide`, `OrderType`, `OrderStatus` enums; `OrderIntent`, `OrderResult`, `AccountSnapshot`, `PositionSnapshot`, `TradeUpdateEvent` dataclasses; six abstract async methods (`submit_order`, `cancel_order`, `get_open_positions`, `get_account`, `get_orders`, `stream_trade_updates`). All broker-facing strategy/orchestrator code must type-annotate against this interface.
- **execution/alpaca_broker.py** — `AlpacaBroker(BrokerBase)` using alpaca-py. Reads `settings.ALPACA_API_KEY/SECRET_KEY/PAPER`. Supports equity market/limit orders and multi-leg options via `OptionLegRequest`. Streams trade updates via `TradingStream`. Dry-run path: logs intent, returns `ACCEPTED` without touching the broker network.
- **execution/kill_switch.py** — File-based global kill switch. `KILL_SWITCH_FILE = settings.OUTPUT_DIR / "KILL_SWITCH"` is the sentinel. `GlobalKillSwitch.is_active()` checks file presence; `activate(reason)` writes atomically (write-then-rename); `deactivate()` removes the file. `KillSwitchActiveError(RuntimeError)` is raised by `OrderManager.submit_order_with_idempotency` BEFORE any pre-trade check when the file exists. CLI: `python -m execution.kill_switch --activate/--deactivate/--status`. `FLATTEN_ON_KILL=true` logs a CRITICAL reminder (automatic flattening is a future extension).
- **execution/risk_gate.py** — Synchronous ten-check pre-trade risk pipeline. `RiskContext` dataclass holds per-call live state (account, open_positions, macro DTO, returns_df, start_of_day_equity, validation_reports, is_premium_sell_strategy, current_prices, timestamp). `PreTradeRiskGate.run_all(intent, context)` short-circuits at the first failure; the tenth check (`max_order_rate_check`) is always last so blocked orders never consume rate-limit budget. Checks in order: (1) max_position_size, (2) portfolio_heat, (3) max_correlation, (4) daily_loss_limit, (5) macro_kill_switch, (6) hmm_regime, (7) stress_scenario, (8) market_hours (NYSE RTH 09:30–16:00 ET via `zoneinfo`), (9) minimum_validation, (10) max_order_rate. Missing context → conservative pass (never blocks on missing data). Thresholds injectable at construction; default to `settings.*` counterparts.
- **execution/order_manager.py** — `OrderManager` sitting between the signal pipeline and the broker adapter. Responsibilities: (1) kill-switch gate (raises `KillSwitchActiveError` BEFORE dedup); (2) deterministic `client_order_id` via SHA-256 of `(strategy_id, symbol, side, qty, 60s-bucket)` for idempotency; (3) `PreTradeRiskGate.run_all()` — returns `ERROR` OrderResult if blocked; (4) single linear-back-off retry on transient broker errors; (5) `reconcile_state(transactions_store)` — compares broker ground truth vs. internal `TransactionsStore`; logs CRITICAL + fires webhook on drift. `make_client_order_id()` is module-level (usable by callers independently). Dry-run: intercepted in `_submit_with_retry` before broker contact.
- **execution/queue_builder.py** — **Tier 8 Robinhood execution bridge.** Emits a GATED, DRY-RUN proposed-order queue (`output/execution_queue.json`) for a Claude Code agent that consumes the Robinhood Trading MCP (the headless pipeline cannot call MCP tools). NEVER contacts a broker. Reuses `OrderIntent`/`PreTradeRiskGate.run_all`/`GlobalKillSwitch`/`make_client_order_id` to gate each intent in dry-run. `emit_execution_queue` returns `None` (writes nothing) when `settings.ROBINHOOD_EXECUTION_MODE=off`; `build_execution_queue` computes `allow_place = mode=="live" AND gate_allowed AND not kill_switch_active AND notional-cap-set` (structurally False otherwise); `gate_intent` fails CLOSED on exception. AST-safe function names (no `place_*`/`submit_order`). **Per-intent `rationale` carries the real operator-facing "why" (2026-07 fix):** each queue intent's `rationale` field is the engine's actual reasoning (`rec.rationale` — e.g. the advisory's plain-English driver paragraph, also written to `state_snapshot.json`), NOT the strategy label; the label moves to its own additive `strategy` key. Previously `rationale` was `str(rec.strategy or rec.rationale)` and the truthy label short-circuited the `or`, silently discarding the reasoning at the queue boundary so every reviewed order showed a bare label. A follow intent's rationale is an honest **ranking** built from real `pilot_holdings` numbers (`Follow:<id> — ranked #N of M by its signal blend (score S → W% target weight, $X target)`), never a fabricated thesis. `_truncate_rationale` word-boundary-truncates at `_RATIONALE_MAX_CHARS`=1200 (sized for a `RATIONALE_VERBOSITY=verbose` multi-section rationale) with an explicit `…` marker — never a silent mid-sentence cut. **Per-intent `strategy_id` override + `sources`/`overridden` keys (2026-07, composer wiring):** `_intent_dict` now reads `intent_strategy_id = str(getattr(rec, "strategy_id", "") or strategy_id)` for both `client_order_id` and `OrderIntent.strategy_id` — a rec with no such attribute (every pre-existing caller: a plain `Recommendation` or `FollowIntent`) falls through to the batch-level `cfg["strategy_id"]` unchanged, but a rec composed by `execution/compose.py` carries its own owner id (`"advisory"`, `"follow-<pilot_id>"` for a single-owner symbol, or `"composed"` when netted across multiple sources) so `client_order_id` stays byte-identical to what that owner's own direct build would have produced. Two new additive output keys, `sources` (`[{source_id, target_notional}]`, always present, `[]` by default) and `overridden` (populated only when `execution/compose.py`'s advisory-wins rule suppressed a follow claim on that symbol — each entry names the source, its own target notional, and its own real rationale). This is the ONLY change to this file for the composer — see `execution/compose.py` below. See the Tier 8 section below.
- **execution/compose.py** — **cross-Pilot + advisory execution-queue COMPOSER — the single writer of `output/execution_queue.json`.** Two writers previously shared that one file (`main.py`'s advisory cycle, every cycle; `pilots/mirror.py::plan_follow`, request-driven) with no coordination — whichever ran last silently overwrote the other (latent only because the default `ROBINHOOD_EXECUTION_MODE="off"` makes the follow path write nothing). Now every source (advisory, and each actively-followed Pilot) writes its own small JSON file to `output/queue_sources/<source_id>.json` (`{schema_version, source_id, generated_at, targets, dropped_targets}`, atomic write-then-rename) — `write_advisory_source(recommendations)` (raw, unfiltered actionable BUY/SELL recs; conviction filtering happens at COMPOSE time against the live `queue_builder.CONFIG["min_conviction"]`, not write time) and `write_follow_source(pilot, amount, snapshot, prior_mirrored=...)` (`pilots.mirror.build_follow_targets`'s pure current targets, plus any `prior_mirrored` row whose symbol just dropped out — a just-detected drop this follow may still need to force-exit). `compose_and_emit(account_snapshot, ...)` reads every source, calls `compose_targets`, and hands the result to the UNCHANGED `execution.queue_builder.emit_execution_queue` as a `FollowRunResult`-shaped `_ComposedRunResult` — zero `queue_builder.py` changes beyond the `strategy_id`/`sources`/`overridden` additions documented above. **Composition rules**, per symbol: (1) **advisory always wins outright** when present (BUY or SELL, above the conviction gate) — its own recommendation is emitted byte-identical to advisory-alone, any follow claim on that symbol is recorded in the emitted intent's `overridden` field (never netted in) — a deliberate, conservative product decision (confirmed with the operator) since summing an independent risk signal with a preference risks stacking into an oversized position; (2) absent an advisory claim, **follow sources are netted by summing TARGETS, never already-netted deltas** — two Pilots wanting $3750/$2000 of a symbol nets to one correctly-sized trim against actual current holdings, computed exactly once; summing each Pilot's own already-netted delta double-subtracts the same holding (verified in `tests/test_compose.py::TestMoneyTestNetTargetsNotDeltas` against numbers where the sum-of-deltas answer isn't even a valid sell size); (3) the no-trade band scales with the NET target (`max($1, 5% × net_target)`) so a 5% band never suppresses a genuine full drop (`net_target=0` → band=$1 exactly); (4) **force-exit under the union is capped by `source_claim`** — the sum, across every follow source, of whichever of its current-target or dropped-target row exists for that symbol — which is what stops a follow's force-exit from selling the operator's own non-follow-attributed shares in the same name. **Dead-letter posture is asymmetric by design:** a MISSING source (a Pilot never explicitly followed via `plan_follow` yet) is legitimate and excluded from the union, composition proceeds; a CORRUPT (unparseable) or STALE (`generated_at` older than `settings.QUEUE_SOURCE_MAX_AGE_SECONDS`, default 7 days — a documented judgment call, not a derived constant) source instead refuses the ENTIRE compose call (writes nothing, leaves the previously-emitted queue in place) rather than silently composing on a subset of trustworthy data — CONSTRAINT #6 ("don't crash") is not the same promise as "proceed with best-effort partial data," and for order-adjacent code silently proceeding on wrong data is the worse failure mode. `pilots/mirror.py::build_follow_intents` is now a thin single-source wrapper over this module's `compose_targets` (verified byte-identical against all 41 pre-existing tests in `tests/test_pilots_mirror.py`, unchanged — the safety net the whole refactor leans on). Lives inside the sanctioned `execution/` order-code zone; added to `tests/test_pipeline_smoke.py::TestNoOrderFunctions`'s manual `_EXECUTION_ZONE_GUARDED_FILES` scan-target list (a new `execution/` file is unscanned by either guard until added there — see that test's own docstring). Never contacts a broker; defines no order-submission function. **New setting:** `settings.QUEUE_SOURCE_MAX_AGE_SECONDS` (float, default 604800.0 = 7 days).
