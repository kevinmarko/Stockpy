# Optional heavy forecasting dependencies — NOT installed by CI or the base
# `./setup.sh`. Install locally with `./setup.sh --optional` or
# `./.venv/bin/pip install -r requirements-optional.txt`.
#
# What this enables
# -----------------
# TensorFlow activates the hybrid CNN-LSTM forecaster in forecasting_engine.py
# (guarded by TENSORFLOW_AVAILABLE). Without it, run_cnn_lstm_forecast() returns
# zeros and CNN-LSTM silently contributes nothing to the multi-model ensemble
# blend or the inverse-RMSE skill tracker (forecasting/forecast_tracker.py) —
# exactly the graceful-degradation pattern Prophet uses. Everything else in the
# platform runs identically whether or not this file is installed.
#
# Why it is separate from requirements.txt
# ----------------------------------------
# TensorFlow is a ~500MB install and its CNN-LSTM path is a local, opt-in model.
# Keeping it out of requirements.txt keeps CI lean (CI installs requirements.txt
# only) and leaves the two TF-dormant tests fast.
#
# numpy safety (load-bearing — do not weaken)
# -------------------------------------------
# requirements.txt pins `numpy>=2.0,<2.3` (the <2.3 ceiling is enforced by
# numba==0.61.2 via pandas-ta/vectorbt). TensorFlow < 2.19 caps numpy below 2.2
# and would DOWNGRADE it, breaking numba's JIT paths. TF >= 2.19 accepts numpy
# 2.2.x, so the floor below keeps numpy exactly where requirements.txt wants it.
# Verified via `pip install --dry-run`: TF 2.21 leaves numpy 2.2.6 untouched and
# adds only net-new packages (keras, ml_dtypes, h5py, wrapt, …) — no downgrades.
#
# Platform note: on Apple Silicon macOS the plain `tensorflow` package ships
# native arm64 CPU wheels — do NOT use the deprecated `tensorflow-macos`, and
# `tensorflow-metal` (GPU) is unnecessary for this tiny model and adds version
# churn, so it is intentionally omitted.
tensorflow>=2.19

# CPU-only PyTorch — activates real local ProsusAI/FinBERT inference in
# signals/news_catalyst.py (see _get_finbert_pipeline / score_headlines)
# instead of the built-in keyword-lexicon fallback. requirements.txt already
# pins `transformers>=4.35.0`, but transformers needs a PyTorch OR
# TensorFlow backend to actually run a model -- without one, FinBERT
# silently falls back to the lexicon on every install, which is why this
# lives here rather than in requirements.txt. `torch>=2.0` is a broad,
# CPU-friendly floor (this repo has no CUDA/GPU requirement -- FinBERT is a
# small BERT-base model, CPU inference is fast enough for a batched,
# per-cycle headline-scoring workload); pip resolves the correct CPU wheel
# for the host platform automatically for a plain `torch` install (no
# `--index-url` / `+cpu` suffix needed on macOS or Linux x86_64/arm64).
# Everything else in the platform runs identically whether or not this file
# is installed -- see FINBERT_ENABLED / FINBERT_SCORE_CACHE_ENABLED in
# settings.py for the graceful-degradation contract.
torch>=2.0

# faiss-cpu activates the embedded local vector store used by the RAG-Powered
# Portfolio Contextualizer (data/rag_index.py, gated behind
# settings.RAG_PORTFOLIO_CONTEXT_ENABLED, default False). It indexes the
# already-ingested sentiment corpus (sentiment_ingestion_audit) into a local
# faiss.IndexIDMap(IndexFlatIP) persisted to output/rag_index/index.faiss —
# a single-operator desktop app has no need for a server-mode vector DB
# (Qdrant/Docker); an embedded, in-process library is the right fit, matching
# this file's own tensorflow opt-in-heavy-dependency precedent. Without it,
# data/rag_index.py's DocumentVectorStore lazy-imports faiss inside every
# method and degrades to a graceful no-op (0 indexed / [] retrieved) exactly
# like the CNN-LSTM path above without tensorflow — the rest of the platform,
# including engine/portfolio_context.py's always-present deterministic
# sector-exposure summary, is unaffected either way.
faiss-cpu>=1.8.0
